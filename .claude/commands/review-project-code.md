# /review-project-code — 项目代码审查

你是一名严格执行代码审查的高级工程师。对当前变更进行逐项审查，输出结构化报告。

## 审查流程

### 第一步：收集信息

1. 读取变更范围：`git diff HEAD`（未提交）或 `git diff main...HEAD`（分支）
2. 确认变更涉及的模块：`src/parser/` | `src/api/` | `src/formatter/` | `src/writer/` | `src/cli/` | `src/errors.ts`
3. 检查是否有对应的测试文件变更

### 第二步：逐项审查

按以下检查清单逐项审查，每项给出 ✅ 通过 / ⚠️ 警告 / ❌ 违规 的判定。

---

#### 一、类型安全（宪法第二条 — 不可协商）

- [ ] **TS-1**: `tsconfig.json` 中 `strict: true` 是否生效（运行 `pnpm exec tsc --noEmit` 确认零错误）
- [ ] **TS-2**: 变更中是否存在 `any` 类型（包括隐式 `any`，即缺少类型注解的参数）
- [ ] **TS-3**: 所有新增函数参数和返回值是否有显式类型定义
- [ ] **TS-4**: 是否存在 `as` 类型断言（除非有注释说明原因）
- [ ] **TS-5**: 对象结构是否使用 `interface` 定义，联合类型/工具类型是否使用 `type`

#### 二、测试覆盖（宪法第三条 — 不可协商）

- [ ] **T-1**: 变更是否包含对应的测试（新功能 → 新测试，Bug修复 → 回归测试）
- [ ] **T-2**: 运行 `pnpm test -- --run` 是否全部通过
- [ ] **T-3**: 测试是否测试行为而非实现细节（不依赖内部变量名、CSS 类名、DOM 结构）
- [ ] **T-4**: 错误路径是否有测试覆盖（错误类、退出码、边界条件）

#### 三、错误处理（宪法第四条 — 不可协商）

- [ ] **E-1**: 所有异步操作是否使用 `try/catch` 或 `.catch()` 处理错误
- [ ] **E-2**: 是否存在静默吞掉异常的情况（空 catch 块、未处理的 Promise）
- [ ] **E-3**: 错误是否使用项目自定义错误类（`ParseError` / `FetchError` / `NetworkError` / `WriteError` / `RateLimitError`）
- [ ] **E-4**: CLI 层的退出码映射是否正确（1=无URL, 2=解析错误, 3=404, 4=网络, 5=写入, 6=限流）

#### 四、简单性（宪法第一条）

- [ ] **S-1**: 是否引入了非必需的依赖
- [ ] **S-2**: 是否存在过度抽象（不必要的 HOC、工厂模式、策略模式等）
- [ ] **S-3**: 是否有 YAGNI 违规（为假设的未来需求预留扩展点）
- [ ] **S-4**: 能否用更简单的方式实现相同功能（三行相似代码优于一个过早的抽象）

#### 五、明确性（宪法第四条）

- [ ] **C-1**: 命名是否自解释（无 `data`, `info`, `item`, `obj`, `result` 等模糊名称）
- [ ] **C-2**: 布尔值是否使用 `is`/`has`/`should` 前缀
- [ ] **C-3**: 状态传递是否显式（通过参数/返回值，非全局/模块级变量）
- [ ] **C-4**: 导出函数是否有清晰的单一职责

#### 六、架构与分层

- [ ] **A-1**: API 层（网络请求）与格式化层（Markdown 生成）是否分离
- [ ] **A-2**: CLI 层是否仅做参数解析和流程编排，不含业务逻辑
- [ ] **A-3**: 模块间是否通过类型接口通信（非直接访问对方内部实现）
- [ ] **A-4**: `process.exit()` 是否仅在 CLI 顶层 `run()` 中调用

#### 七、规范对齐

- [ ] **R-1**: 是否符合 `specs/` 下的规格说明（若存在对应 spec）
- [ ] **R-2**: 输出格式是否符合 spec 中定义的 Markdown 结构
- [ ] **R-3**: Git commit message 是否遵循 Conventional Commits 格式

---

### 第三步：输出报告

使用以下格式输出：

```
## Code Review Report

**变更范围**: [涉及的文件列表]
**审查结果**: [通过 / 需修改] (X/Y 项通过)

### 检查结果

| 编号 | 检查项 | 结果 | 备注 |
|------|--------|------|------|
| TS-1 | 类型检查 | ✅/⚠️/❌ | ... |
| ... | ... | ... | ... |

### 必须修复 (❌)
[列出所有违规项，附文件名和行号]

### 建议改进 (⚠️)
[列出所有警告项]

### 亮点
[值得肯定的实现方式]
```

### 规则

- **❌ 违规项为阻塞性问题**，必须修复后才能合并
- **⚠️ 警告项为建议性改进**，不阻塞但应考虑
- 重点关注宪法中标注「不可协商」的条款（类型安全、测试先行、错误处理）
- 审查时如果发现 spec 中未覆盖但实际需要处理的场景，记录为建议
