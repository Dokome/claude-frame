# ClaudeFrame 项目开发宪法
# Version: 1.0, Ratified: 2026-05-01

本文件定义了本项目不可动摇的核心开发原则。所有AI Agent在进行技术规划和代码实现时，必须无条件遵循。

---

## 第一条：简单性原则 (Simplicity First)
**核心：** 只做必要的事。绝不进行不必要的抽象，绝不引入非必需的依赖。
- **1.1 (YAGNI):** 只实现明确要求的功能，不为假设的未来需求预留扩展点。
- **1.2 (内置优先):** 优先使用浏览器/平台内置 API 和 React 内置能力（如 `useState`, `useEffect`, `useMemo`），避免为 React 已原生支持的功能引入第三方库。
- **1.3 (反过度工程):** 简单的函数和组件优于复杂的 HOC、render props 或过度抽象的自定义 Hook。三个相似场景优于一个过早的抽象。

---

## 第二条：类型安全铁律 (Type Safety Imperative) - 不可协商
**核心：** TypeScript 是本项目的基石，类型安全不可妥协。
- **2.1 (严格模式):** 必须启用 `strict: true`。禁止使用 `any`，必要时使用 `unknown` 并进行类型收窄。
- **2.2 (先定义类型):** 所有函数参数、返回值和组件 Props 必须有显式类型定义。优先使用 `interface` 定义对象结构，`type` 用于联合类型和工具类型。
- **2.3 (杜绝类型断言):** 禁止使用 `as` 类型断言绕过类型检查，除非在确信安全的边界处（如第三方库的类型不完整），且必须附带注释说明原因。

---

## 第三条：测试先行铁律 (Test-First Imperative) - 不可协商
**核心：** 所有新功能或 Bug 修复，都必须从编写一个（或多个）失败的测试开始。
- **3.1 (TDD 循环):** 严格遵循 "Red-Green-Refactor" 循环。
- **3.2 (组件测试优先):** 使用 React Testing Library 进行组件行为测试，测试用户交互而非实现细节。优先使用 `screen.getByRole` 等无障碍查询。
- **3.3 (拒绝实现耦合):** 测试不得依赖组件内部状态变量名、CSS 类名或 DOM 结构。测试应从用户视角出发：渲染 → 交互 → 断言结果。

---

## 第四条：明确性原则 (Clarity and Explicitness)
**核心：** 代码的首要目的是让人类易于理解。
- **4.1 (错误处理):** **不可协商**：所有可能的运行时错误都必须被显式处理。异步操作必须使用 `try/catch` 或 `.catch()` 处理错误，禁止静默吞掉异常。
- **4.2 (无隐式状态):** 组件间的状态传递必须显式（通过 Props 或 Context），禁止通过全局变量、模块级变量或 DOM 属性隐式传递状态。
- **4.3 (命名即文档):** 变量、函数和组件的命名必须自解释。禁止使用 `data`, `info`, `item` 等模糊名称。布尔值使用 `is`/`has`/`should` 前缀，事件处理使用 `on` 前缀。

---

## 治理 (Governance)
本宪法具有最高优先级，其效力高于任何 `CLAUDE.md` 或单次会话中的指令。
